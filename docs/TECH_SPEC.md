# Technical Specification — 3-Mile Drive Map

## 1. Architecture Overview

```
┌─────────────────┐       ┌─────────────────┐       ┌──────────────────┐
│   React + TS    │──────▶│   FastAPI        │──────▶│  OpenRouteService│
│   MapLibre GL   │◀──────│   (API shield +  │◀──────│  (Isochrones +   │
│   Frontend      │       │    cache layer)  │       │   Directions)    │
└─────────────────┘       └─────────────────┘       └──────────────────┘
```

**Frontend:** React + TypeScript + MapLibre GL JS
**Backend:** FastAPI — acts as API-key shield and cache layer
**Routing provider:** OpenRouteService (ORS) — primary
**Fallback provider:** Self-hosted Valhalla (escape hatch, not MVP)

### Why this architecture (not pre-computed)

The tap-to-check routing feature requires a live routing API for every destination click. Since we already need ORS for directions, we use it for the isochrone polygon too — one provider, one integration. The polygon is cached aggressively so the API cost is minimal.

A pre-computed OSMnx approach remains viable as a **fallback for the polygon only** — if ORS goes down, a cached GeoJSON file generated offline can serve as the service area visual. But the route-check feature always needs a live API.

## 2. Provider Configuration (ORS)

### Isochrones endpoint
```
POST https://api.openrouteservice.org/v2/isochrones/driving-car
Headers: Authorization: {ORS_API_KEY}
Body: {
  "locations": [[-105.9384, 35.6824]],
  "range": [4828],
  "range_type": "distance",
  "units": "m",
  "smoothing": 25
}
```

- `range_type: "distance"` — critical. This computes equidistant (isodistance), not isochrone.
- `smoothing` — reduces polygon jaggedness. Tune between 0-100.
- Profile: `driving-car`

### Directions endpoint
```
POST https://api.openrouteservice.org/v2/directions/driving-car/geojson
Headers: Authorization: {ORS_API_KEY}
Body: {
  "coordinates": [[-105.9384, 35.6824], [dest_lon, dest_lat]],
  "preference": "shortest"
}
```

- `preference: "shortest"` — **critical**. Without this, ORS returns fastest-time route which may be longer mileage, violating the 3-mile distance policy.
- Response includes `summary.distance` (meters) for the within-limit check.

### Important: polygon ≠ route (isochrones)

ORS isochrone and directions use different algorithms internally. The polygon is a generalized estimate; the point-to-point route is exact. A location near the polygon boundary might be inside the polygon but > 3 miles by route, or vice versa. **The route check is authoritative.** The UI must handle this gracefully.

### Route-based polygon (accurate boundary)

To align the shaded boundary with the same routing engine as tap-to-check, a **route-based polygon** can be generated by running route checks along 24 radial spokes from the origin and finding the 3-mile boundary point on each. This uses the directions API with `preference="shortest"` only, so the polygon and tap-to-check verdict match.

- **Generation:** Run `python scripts/generate_route_polygon.py` from project root (requires `ORS_API_KEY`). Writes `cache/area_3mi_route.geojson`. ~240 ORS calls per run.
- **Serving:** `GET /api/area?miles=3` returns the route-based polygon if that file (or in-memory cache) exists; otherwise falls back to ORS isochrones.
- **Trigger:** Manual or cron (e.g. weekly). Not computed on-demand to avoid long first-request latency and API cost.

## 3. Backend API Contract

### `GET /api/config`

Returns origin metadata for frontend initialization.

```json
{
  "hotel_name": "New Mexico State Capitol",
  "address": "411 South Capitol St, Santa Fe, NM 87501",
  "coordinates": [-105.9384, 35.6824],
  "default_miles": 3,
  "max_miles": 5
}
```

### `GET /api/area?miles=3`

Returns cached GeoJSON FeatureCollection (Polygon/MultiPolygon) representing the drivable service area.

**Cache strategy:**
- **Route-based (preferred):** If `cache/area_3mi_route.geojson` exists (or in-memory key `area_3mi_route`), return it. No TTL; regenerate via `scripts/generate_route_polygon.py`.
- **Isochrone fallback:** Key `f"area_{miles}_{profile}"`, TTL 24 hours, in-memory + file `./cache/area_{miles}_{profile}.geojson`. On miss: call ORS isochrones, cache, return.

```json
{
  "type": "FeatureCollection",
  "features": [{
    "type": "Feature",
    "geometry": { "type": "Polygon", "coordinates": [...] },
    "properties": {
      "distance_miles": 3.0,
      "distance_meters": 4828,
      "computed_at": "2026-02-10T12:00:00Z"
    }
  }]
}
```

### `GET /api/route?to=-105.99,35.68`

Computes shortest-distance route from origin to destination.

```json
{
  "route": {
    "type": "Feature",
    "geometry": { "type": "LineString", "coordinates": [...] },
    "properties": {}
  },
  "distance_meters": 4200.5,
  "distance_miles": 2.61,
  "duration_seconds": 480,
  "within_limit": true,
  "limit_miles": 3.0
}
```

**Error responses:**
- `400` — invalid coordinates
- `404` — no route found (destination unreachable by road)
- `502` — ORS upstream error
- `429` — rate limited

## 4. Frontend Rendering

### Map layers (bottom to top)

1. **Base tiles** — muted, desaturated. Candidates:
   - Stadia Stamen Toner Lite (free, clean)
   - MapTiler Streets (free tier, good carto)
   - Protomaps PMTiles (self-hosted, fully open)
2. **Service area polygon** — semi-transparent fill + dashed outline stroke
3. **Route line** — drawn on click, removed on next click or reset
4. **Origin marker** — prominent icon with popup
5. **Destination marker** — appears on click with verdict badge

### Interaction flow

```
1. Page loads → fetch /api/config → center map → fetch /api/area?miles=3 → render polygon
2. User clicks map → fetch /api/route?to=lon,lat → render route line + verdict panel
3. User clicks again → clear previous route → fetch new route
4. User clicks "Reset" → clear route, recenter on origin
```

## 9. Escape Hatches

### If ORS goes down or hits limits
1. **Polygon:** Serve from file cache (`cache/area_3mi.geojson`). Generate offline with `scripts/generate_fallback.py` using OSMnx.
2. **Routing:** Temporarily disable tap-to-check, show polygon only with message "Route checking temporarily unavailable."

### If ORS doesn't cut it long-term
Self-host **Valhalla** with New Mexico OSM extract. Valhalla supports distance-based isochrones (`contours` with `distance` flag) and shortest-distance routing. Heavier ops but zero API dependency.

### Fallback polygon generator (OSMnx)

The `scripts/generate_fallback.py` script provides an offline-computed polygon independent of any API:

```python
# Pseudocode — full implementation in the script
import osmnx as ox
import networkx as nx

G = ox.graph_from_point((35.6824, -105.9384), dist=8000, network_type='drive')
origin_node = ox.nearest_nodes(G, -105.9384, 35.6824)
distances = nx.single_source_dijkstra_path_length(G, hotel_node, cutoff=4828, weight='length')
# → concave hull of reachable nodes → export GeoJSON
```

This can regenerate the polygon quarterly or whenever the road network changes significantly.
